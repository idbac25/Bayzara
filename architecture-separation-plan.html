1. Executive Summary

Currently Bayzara is built as a monolith — the frontend (what users see) and backend (business logic, data access) are mixed together in one Next.js application. This works fine for a web app, but it creates a wall when you want to add a mobile app, third-party integrations, or a separate admin panel.

The CTO's recommendation is to adopt an API-First architecture — build a dedicated backend API that speaks a universal language (REST or GraphQL) that any client can connect to: the web app, the mobile app, a desktop app, or even third-party tools.

Simple analogy: Right now, the kitchen and the dining room are the same room. The CTO wants to build a proper kitchen (backend API) with a serving hatch — the web app and mobile app both order food through the same hatch, getting the same menu, the same quality, from one place.

In the current setup, the Next.js app does three jobs at once:

  • Renders pages — converts data into HTML for the browser
  • Runs business logic — invoice creation, EVC matching, user access control
  • Accesses the database — queries Supabase directly from Server Components

This is common and perfectly valid for a web-only app. The limitation appears the moment you want a second client (mobile app) to use the same data and logic.

3. The Problem With the Current Approach

Problem
Mobile app has nowhere to connect
A React Native or Flutter mobile app cannot run Next.js Server Components. It needs a URL-based API (REST or GraphQL) to call. Currently that doesn't exist as a standalone service.
Problem
Business logic is tied to the web UI
Logic like "auto-match EVC payment to invoice" or "generate document number" lives inside Next.js API routes that are tightly coupled to the web deployment. Changes to logic require redeploying the whole frontend.
Problem
Cannot scale independently
If API traffic spikes (e.g. lots of EVC syncing), you cannot scale just the API — you would have to scale the entire Next.js app including all the HTML rendering.
Problem
Hard for third-party integrations
If a partner wants to connect their system to Bayzara's data (e.g. an accounting app, a bank), there is no clean API endpoint they can call — they would have to scrape the web app.

Authentication Flow on Mobile

1
User logs in on the mobile app
Mobile app calls Supabase Auth directly (using the Supabase JS SDK). Supabase returns a JWT access token and refresh token.
2
Token is stored securely on the device
Stored in the device's secure keychain (not plain storage). The Supabase SDK handles this automatically.
3
Every API request includes the token
Mobile app sends Authorization: Bearer [token] with every request to the Bayzara backend API.
4
Backend verifies the token
The Bayzara API verifies the JWT against Supabase's public key. If valid, it extracts the user ID and processes the request. No separate mobile login system needed.
5
Token auto-refreshes
The Supabase SDK silently refreshes expired tokens in the background. The user stays logged in without interruption.

• Web app never touches service role key
• Frontend and backend deployed independently

What the Web App Becomes

The Next.js app is stripped down to a pure frontend. It is responsible for:

  • Rendering pages and components
  • Handling user interactions (clicks, form submissions)
  • Calling the backend API to read and write data
  • Managing UI state (loading, errors, modals)

It is NOT responsible for database access, business rules, or secrets. It becomes a "dumb" client — smart on the screen, but relying entirely on the API for data.

8. What Stays the Same (Supabase)

Importantly, Supabase does not go away. It remains the core data layer. The key change is that only the backend API talks to Supabase directly — clients no longer query Supabase themselves.

9. Migration Roadmap — Phase by Phase

This does not need to happen all at once. The recommended approach is incremental — keep the app working at every step.

Phase 1 — Setup the API project (1–2 weeks)
Create a new repository bayzara-api. Set up Hono.js with Cloudflare Workers. Configure connection to existing Supabase database. Implement JWT middleware for auth. Deploy a "hello world" endpoint. No changes to the web app yet.
Phase 2 — Migrate core API routes (2–3 weeks)
Move the existing Next.js API routes (/api/business/create, /api/evc/connect, /api/evc/activate) into the new API. Update the web app to call the new API URLs instead. Test thoroughly. This is the most critical phase.
Phase 3 — Add full CRUD endpoints (3–4 weeks)
Build REST endpoints for all modules: invoices, clients, vendors, inventory, leads, reports. The web app's Server Components are refactored to call these endpoints instead of querying Supabase directly. The service role key moves to the API only.
Phase 4 — Mobile app development (6–10 weeks)
Start building the React Native mobile app. Since the API is already fully built and tested by the web app, the mobile app just needs to call the same endpoints. Authentication works out of the box via Supabase JWT. Focus on mobile-specific UX.
Phase 5 — API versioning & documentation (1 week)
Add API versioning (/v1/ prefix). Generate API documentation using OpenAPI/Swagger. This makes it easy to onboard new developers, build third-party integrations, and maintain the API as it grows.
